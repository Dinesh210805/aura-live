# VLM Selector (Set-of-Marks Element Selection)

**File**: `perception/vlm_selector.py`

> `VLMSelector` is **not an agent**. It is Layer 3 of the three-layer perception pipeline,
> called internally by `PerceiverAgent` via `PerceptionController`. There is no `agents/visual_locator.py`.

---

## Role

`VLMSelector` implements the **Set-of-Marks (SoM)** approach to UI element selection. Given a screenshot with labeled bounding boxes from the CV detector (OmniParser/YOLOv8), the VLM selects which box corresponds to the user's target.

**Critical invariant**: The VLM NEVER generates pixel coordinates. It only selects a label letter from the provided candidates. Coordinates always come from the bounding box center of the selected Detection object.

---

## SelectionResult Dataclass

```python
@dataclass
class SelectionResult:
    success: bool
    selected_id: Optional[str]          # Label letter: "A", "B", "C", etc.
    detection: Optional[Detection]       # Full CV detection data
    coordinates: Optional[Tuple[int, int]]  # Center of selected bounding box
    confidence: float
    reasoning: str                       # Debug explanation
    screen_description: str              # VLM description of what's on screen
    source: str = "vlm_selector"
    latency_ms: float
```

---

## Selection Prompt

The VLM receives the screenshot with labeled regions and this structured prompt:

```
You are analyzing a mobile app screenshot with labeled UI regions.
Each region is marked with a large letter (e.g., A, B, C) inside a red bounding box.

Labeled regions on screen:
{regions_list}    ← description of each labeled region

User wants to interact with: "{intent}"

Step 1 — In one sentence, describe what screen/app you see and which region looks like the target.
Step 2 — Output JSON:
{"thinking": "I see X. Region Y looks like target because Z.", "label": "Y", "description": "..."}

Use NONE as the label if no region matches.
```

The `regions_list` is generated by the CV detector — each label A, B, C... maps to a detected bounding box with coordinates already known.

---

## Why This Eliminates Spatial Hallucination

Without SoM, a VLM prompted to "tap the Liked Songs button" might output coordinates like `(540, 823)`. VLMs are poor at precise spatial prediction — they can describe what's on screen semantically but not where exactly.

With SoM:
1. YOLOv8 detects all likely interactive regions (geometrically valid, has width/height)
2. Each region gets a letter label drawn on the screenshot
3. The VLM only needs to say "region B" — it doesn't need to produce coordinates
4. The coordinates come from the CV detector's bounding box center

---

## `select_with_fallback(intent, candidates, screenshot)`

The main entry point. Tries the primary VLM (Groq Scout). If selection fails or returns `NONE`:
1. Retries with alternate prompt
2. Falls back to Gemini VLM
3. Returns `SelectionResult(success=False)` if all fail

The `VLMTimeout` (Fix G6): this is wrapped in a `ThreadPoolExecutor` call by the perception pipeline with a 30s timeout (configurable via `settings.vlm_timeout_seconds`).

---

## Integration with Perception Pipeline

`VLMSelector` is Layer 3 of the three-layer pipeline:
```
Layer 1: UI Tree Parser → finds accessible elements
Layer 2: OmniParser (YOLOv8) → detects visual regions, generates labeled screenshot
Layer 3: VLMSelector → chooses which labeled region matches intent
Layer 4: Heuristic → fallback using cached detections
```

The result's `coordinates` field is the center of the selected bounding box from Layer 2.

See [`pipeline.md`](pipeline.md) for the full pipeline flow including timeout handling.
