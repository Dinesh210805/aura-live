🤖 AURA — Agent Flow & Memory Map

Complete walk-through of a single command: "open Amazon, search for iPhone 17 Pro and add it to my cart"

1 · Agent Pipeline Overview
① Commander
Parses user text → structured intent dict
→
② Planner
Creates Goal + phases[] skeleton
→
③ Coordinator loop
Drives execution. Picks next subgoal via ReactiveStep LLM
→
④ PerceiverAgent
Takes screenshot, reads UI tree, calls ScreenReader VLM
→
⑤ VisualLocator
Finds pixel coordinates for a tap/swipe target
→
⑥ ActorAgent
Dispatches gesture to device via ADB/adapter
→
⑦ VerifierAgent
Checks screen changed, success criteria passed
→
⑧ Responder
Speaks final result to user
How phases and subgoals relate
Phases = high-level skeleton created once: ["Open Amazon", "Search for iPhone 17 Pro", "Add first result to cart"]
Subgoals = concrete one-action steps generated reactively per-screen as each phase executes.
The Coordinator loop: observe screen → ask LLM "what single action next?" → execute → verify → repeat.

Key Design Insight
The reactive LLM is stateless per call — it only knows what you give it. Every piece of context it needs (what's on screen, what's been done, what phase we're in, what still needs to happen) must be explicitly assembled from the memory structures above and injected into the prompt. The agent's "intelligence" is the sum of its memory plumbing.
The loop bug explained in one line
BLOCKER was only watching for overlapping UI blockers (popups). It was not watching whether the current step's target was reachable — so the reactive LLM acted into the void. The fix makes the VLM use its CURRENT STEP mission context to proactively test reachability before saying "BLOCKER: None".
5 · Context Gaps — Where Needed Memory Is NOT Shared
How to read this table
These are real gaps found by tracing every step_history, screen_description, plan_context, and pre_action_context reference through all agent files. "Severity" = how much decision quality degrades when the field is absent.
Sev. Gap Where What's Missing / Why It Matters What Already Exists to Fill It
CRITICAL step_history silently discarded in verifier verifier_agent.py
verify()		 step_history is accepted as a parameter but never read inside the function body. Not passed to _check_success_criteria(), _vlm_verify_commit(), or _poll_for_change(). The verifier is flying blind — it sees the post-action screen but has no context about what was attempted before. step_history is already passed in by Coordinator. Just wire it through to _vlm_verify_commit().
CRITICAL pre_action_context["cart_count"] never reaches the VLM verifier verifier_agent.py
_vlm_verify_commit()		 Coordinator captures subgoal.pre_action_context = {"cart_count": 2} before each commit action. _vlm_verify_commit() receives the subgoal object but never reads subgoal.pre_action_context. The VLM confirms "add to cart succeeded" by looking at the post-action screenshot alone — it cannot compare cart count before vs after. subgoal.pre_action_context is already set on the object passed in. Read it and include in the VLM's verify prompt: "Cart count before action: 2. Confirm it changed."
HIGH running_screen_context degrades to 8 element labels coordinator.py →
reactive_step_generator.py		 When a loop iteration skips the VLM describe-screen call (e.g. fast-path actions), running_screen_context is set to 8 UI element labels joined by ;: "Search; Cart; Account; Home". The ReactiveStep LLM then receives this as its sole screen context. It cannot reason about current app state and may generate incorrect steps. StepMemory.screen_description from the most recent entry that has a non-null value. Fall back to last VLM description rather than raw labels.
HIGH _recent_steps strips result + screen_description per step perceiver_agent.py
_build_recent_steps()		 _recent_steps is built as "[tap] Tap search bar → [type] iPhone..." — just action+description, truncated at 50 chars. Loses: result (✅/❌), screen_type, and screen_description. So the screen reader VLM sees "steps done" but not whether they worked or what the screen looked like after each one. Include result emoji in the step string: "[tap] ✅ Tap search bar → [type] ❌ Type failed". The screen reader can then tailor BLOCKER reasoning to repeated failures.
HIGH OmniParser fallback missing user_command / plan_context / subgoal visual_locator.py
locate_from_bundle()		 Primary path (locate_with_annotated_ui_tree) receives full context: user_command, plan_context, subgoal_description included in a FULL PLAN: block. OmniParser fallback receives only element_description + screen_context. When the primary path fails (element not in accessibility tree) and the fallback fires, the VLM picks from CV-detected candidates with much less grounding — exactly the scenario that causes wrong-element taps. All three fields are in scope at the call site in perceiver_agent.py line 218. Pass them to locate_from_bundle() and thread to _legacy_locate_from_bundle().
MEDIUM _format_history() drops screen_description per step reactive_step_generator.py
_format_history()		 History passed to the reactive LLM shows: "✅ tap(Library) → native → ✅ scroll_down(None)". It does NOT include what the screen looked like after each step. After a failed "Add to Cart" tap, the LLM sees "❌ tap(Add to Cart) → webview" but not "…and the screen showed the home page" — it cannot reason about what went wrong. StepMemory.screen_description is stored. For result=failed entries, append a 1-line summary: "→ screen: {description[:80]}".
MEDIUM plan_context never reaches screen_reader.describe_screen() perceiver_agent.py →
screen_reader.py		 perceiver.perceive() receives plan_context and passes it to locate_with_annotated_ui_tree() — but NOT to perception_controller.request_perception() which is the path to describe_screen(). Screen reader knows current step and last 3 steps but not what comes next in the plan. Knowing "next step will be Add to Cart" would let the VLM confirm whether it's visible sooner. Add plan_context parameter to request_perception() → describe_screen() and inject into AGENT MISSION block as UPCOMING STEPS.
LOW No last_failure_reason in reactive step prompt prompts/reactive_step.py The steps_done history shows ❌ markers but gives no reason (e.g. "element not found", "screen didn't change", "verifier returned False"). The LLM may retry the same failed approach without knowing the failure modality. Add a last_failure_reason template field. Populate from StepMemory.result + subgoal.last_error if the most recent step failed.
6 · Multimodal LLM — What Changes (and What It Unlocks)
What "multimodal" means here
The VLMs used in AURA (e.g. llama-4-scout, gemma3-27b-it) are full multimodal LLMs — they accept images and text together, and can reason across both simultaneously, exactly like a normal text LLM but with vision added. This is distinct from a pure vision model. It means every place where AURA currently passes a text description of a screen, it could instead (or additionally) pass the raw screenshot and let the LLM see directly.
🔀 Collapse ScreenReader + VisualLocator into one call −1 VLM CALL per action
Current: 2 separate VLM calls per action cycle:
  • VLM call 1 → describe_screen() → returns 7-section text description
  • VLM call 2 → locate_with_annotated_ui_tree() → returns element ID + coordinates
With multimodal: One combined call that returns both in a single structured JSON response:
  • screen_description: the 7-section report
  • target_element: element ID from the annotated tree
  • blocker: TARGET NOT VISIBLE / INPUT NOT FOCUSED
Why it works: The LLM sees the image + accessibility tree + subgoal in one prompt and can simultaneously describe the screen and identify the target. It already has all the context — the split is artificial.

⚠ Risk: combined call failure = lose both description AND coordinates (currently separate fallback paths).
📸 Before/After image comparison in VerifierAgent MUCH STRONGER
Current: Verifier calls describe_screen() on the post-action screenshot only. It is text-only guessing from one frame.

With multimodal: Feed both screenshots to the VLM in one call:
  • "Here is the screen BEFORE the action [img1] and AFTER [img2]."
  • "The action was: tap Add to Cart. Did the cart count increase? Did an overlay appear confirming the add?"
  • Also: "Cart count before: {pre_action_context['cart_count']}"
Why it works: Visual delta is far more reliable than reading a single screen. The VLM can detect subtle changes (badge +1, overlay appeared, button changed state) that would not be obvious from post-action description alone.

This directly closes Gap #2 (pre_action_context unused) and Gap #1 (step_history for verifier).
🆘 Direct screenshot fallback when screen_context degrades RECOVERS GAP #3
Current: When running_screen_context degrades to 8 element labels, the ReactiveStep LLM receives near-useless context like "Search; Cart; Account; Home".

With multimodal: Coordinator detects degraded context (length < threshold OR no VLM description in last step), attaches the raw screenshot to the ReactiveStep LLM call, and lets it see the screen directly.

Why it works: The ReactiveStep LLM IS a multimodal LLM — it just isn't being sent images currently. No architecture change needed, just pass the screenshot base64 when the text context is weak.

⚠ Cost: Image tokens (~1200 tokens) added to an already ~1500-token prompt.
🗺 Image-grounded replanning SMARTER REPLAN
Current: planner.replan() receives an obstacle string + text description of the current screen. It rebuilds phases from text alone.

With multimodal: Pass the current screenshot to the replanner LLM:
  • Planner sees that we are on the Amazon Home page (not the product page) and can reason spatially about what went wrong
  • For navigation failures, the image shows exactly where the app ended up — no risk of text description being imprecise
Why it works: Replan decisions ("we're on Home, we need to get back to product page") are inherently visual. The planner currently trusts the text description of where things went wrong — an image would be unambiguous.
⛔ What Multimodal Does NOT Change NO BENEFIT
Commander (intent parsing): Pure text task. No image at this stage. Multimodal adds nothing.
Planner (goal decomposition): Works from intent + app list. No screen to look at yet.
Validator: Pure rule-based, no LLM at all. Not affected.
ActorAgent: Executes gestures via ADB. No LLM in the hot path.
Responder: Reads from TaskState.spoken_response. No visual reasoning needed.

The multimodal advantage is concentrated in the perception-to-action loop agents: ScreenReader, VisualLocator, VerifierAgent, and Coordinator's fallback logic.
Biggest single win available
Verifier before/after image comparison — because it closes two critical gaps simultaneously (verifier ignores step_history, pre_action_context unused) and doesn't require changing any other agent. The VLM already accepts images. You just need to save the pre-action screenshot in Coordinator and pass both frames to _vlm_verify_commit().
Cheapest single fix available (code bug, not design gap)
step_history param in verifier_agent.py — it is already accepted as a parameter but never read. Plumbing it through to _vlm_verify_commit() is a 3-line change and turns the verifier from context-blind into context-aware.
